# War Bulletin 6: The Inhabitation Threshold

**Date:** 2026-04-10
**Context:** M-stack-inhabitation (IDENTIFY), M-repl-wins-over-cli (Checkpoint 2),
  M-hypergraph-operator (IDENTIFY). Single extended session: CLI -> REPL migration,
  evidence pipeline, context retrieval, hypergraph alignment, Arxana browser wiring.
**Trigger:** Joe ran `cr new` in an Emacs shell buffer and it hung Emacs. The
  investigation that followed ended up demonstrating that the REPL surface can
  be strictly better than the CLI — and that the evidence generated by inhabiting
  it immediately produces value (pattern retrieval, context certificates, browsable
  conversation history).

## Findings

### Finding 1: Entry Ceremony Is Load-Bearing

The `cr`/`cx` entry scripts had four separate bugs that each individually
bounced users back to CLI: heredoc quoting (`\"` in bash heredocs passes
literal backslash to Emacs), `emacsclient -t` inside Emacs hangs the PTY,
Agency not running requires manual pre-flight, and session resume needs the
correct cwd on the ProcessBuilder.

Once all four were fixed, the REPL became zero-friction from inside Emacs.
The pattern `surface-earns-inhabitation` predicted this precisely: each
friction point is a reason to fall back to the primitive path. Four friction
points = zero inhabitation. Zero friction points = full inhabitation.

**Implication:** Entry ceremony bugs are not UX polish — they are
infrastructure failures. Treat them with the same severity as data loss.

### Finding 2: Ghost Agents Are an I-1 Smell

Repeated `cr new` invocations accumulated ghost agents in the registry
(claude-3 through claude-9, all idle with no session). This is an I-1 smell:
multiple agent identities for what should be one session. The fix (ghost
reclamation in `/agents/auto`) is simple but the diagnostic is important:
if the registry accumulates garbage, the entry ceremony is being retried
rather than succeeding.

**Implication:** Registry hygiene is a liveness signal. Ghost count > 0
means the entry ceremony failed at least once recently.

### Finding 3: Evidence Flows When the Surface Is Inhabited

Within minutes of the REPL becoming inhabitable, evidence started flowing:
chat turns with session IDs, context-retrieval certificates linking turns
to futon3a patterns, and the full conversation became browsable in the
Arxana evidence browser. None of this required new evidence infrastructure —
it was all built and waiting. The missing piece was inhabitation.

This is `inhabitation-feeds-evolution` in action: the Baldwin loop was
starved not because the peripheral was broken, but because nobody was in it.

**Implication:** When evidence flow is zero, check inhabitation before
checking infrastructure. The infrastructure is probably fine.

### Finding 4: Context Retrieval as Cyborg Interoception

Semantic pattern retrieval (futon3a MiniLM embeddings) on each A->B turn,
delivered as desktop notifications with evidence certificates, functions as
interoception for the cyborg loop. The system observes its own coordination
patterns post-hoc. The retrieved patterns are the system's best guess at
what happened in the exchange.

This is the observation phase of the AIF loop. Pre-hoc pattern selection
(predicting which patterns are relevant *before* the turn) requires
accumulating enough post-hoc observations first.

**Implication:** Post-hoc pattern retrieval is not a feature — it is the
training signal for the entire prediction pipeline. Start post-hoc now;
pre-hoc follows from sufficient data.

### Finding 5: The Sorry Topology Is the Stack's Backlog

The JSDQ hypergraph alignment (grand-unified-placemat -> terminal vocabulary
-> hypergraph) produced a sorry topology with 8 typed holes. The candidate
invariant queue has 65 entries. The excursion pipeline has 6 items in mixed
states. These are not three separate backlogs — they are one sorry topology
viewed from three angles (market-facing, infrastructure, exploration).

The M-stack-inhabitation mission treats them as a single portfolio: every
uninhabited surface is a sorry, every sorry has a maintenance cost, and the
portfolio ranks by (cost * probability of closing).

**Implication:** Unify the backlogs. The sorry topology IS the backlog.

### Finding 6: The Evidence Browser Closes the Evidence Loop

The Arxana evidence browser (`arxana://view/evidence-home`) was pointing at
the wrong server (localhost:7070 instead of 47070) and rendered sessions as
server logs rather than conversations. Two fixes — auto-discover the
evidence server, render chat turns as conversation — made the evidence
landscape browsable for the first time.

This closes a loop: evidence is generated by the REPL, logged to the
Agency evidence store, and browsable in Arxana as enriched conversation
history. The loop was designed years ago; it became operational today.

**Implication:** Closing existing loops produces more value than building
new infrastructure. The evidence loop was 90% built; the last 10% was
wiring.

## Strategic Assessment

The stack has crossed an inhabitation threshold. Before today, the primary
coding surface (CLI) generated no evidence. After today, the primary coding
surface (REPL) generates evidence on every turn, retrieves patterns, logs
certificates, and makes the conversation browsable.

The next phase is not building more infrastructure — it is *inhabiting what
exists*. The 65 candidate invariants need triage, not a new queue system.
The excursions need promotion or parking, not more explorations. The
hypergraph needs Clicks and Ticks, not more observables.

The M-stack-inhabitation portfolio is the vehicle. The pattern is clear:
identify the uninhabited surface, reduce its entry friction to zero, inhabit
it, let the evidence flow, and use the evidence to evolve the surface.
Repeat for each surface in the portfolio.

The constraint from the JSDQ analysis applies here too: cargo implies
depositing. The stack has cargo (built infrastructure). The depositing
action is inhabitation. The policy transition is from pi-hermit (build
alone, never deposit) to pi-free-solo (balanced: build, deposit, lay trail,
return). Bristol is the external version; stack inhabitation is the internal
version. Same constraint, same action, different domain.
