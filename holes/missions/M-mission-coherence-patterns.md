## 1. IDENTIFY

### Motivation

The existing mission lifecycle gives a strong procedural scaffold for
running mission-shaped work, but it does not yet provide a compiled
pattern library for what a mission *is*, how it *operates*, and why the
form works so much better than ad-hoc agentic coding or lightweight
spec-writing. The gap is not that missions are absent; the gap is that
their coherence is largely tacit. Joe can use missions effectively, but
the effective structures are not yet extracted into reusable
`flexiarg` patterns.

This matters because missions appear to be a major step-change in the
working economy of agentic coding. HEAD §2 names the live claim:
missions are as much an improvement over agentic coding as agentic
coding is over no-agentic-coding. If that claim is even partly right,
then leaving the practice tacit is a real structural loss. The stack
needs patterns that preserve what missions make possible: high-fidelity
alignment to the operator's intent, co-evolution of prose and code, and
enough discipline that the result is both buildable and communicable to
others.

The mission therefore exists to turn mission-practice from an effective
but partly inarticulate habit into an explicit pattern library. The new
`HEAD` bootstrap in `futon4/holes/mission-lifecycle.md` is part of that
same move: the lifecycle now has a sanctioned place for the
operator-voice intake that this mission needs to theorize.

### Theoretical anchoring

- **Mission as exotype:** HEAD §4's strongest theoretical claim is that
  a mission functions as an *exotype* for a coding agent's operation.
  The mission is not just a prompt or a project brief; it is an
  externalized, evolving form that shapes the agent's trajectory over
  time.
- **Lifecycle plus pattern layer:** `mission-lifecycle.md` gives the
  phase-structure; this mission asks what recurrent design patterns make
  that structure actually effective in practice. The target is a layer
  *above* the lifecycle template, not a replacement for it.
- **Pattern mining from historical missions:** HEAD §3 sets the
  anti-glibness discipline explicitly. The library must be earned by
  real mining from existing mission corpora, not by inventing
  attractive-sounding abstractions. The method note at
  `~/Downloads/1996461.1996523.pdf` is the starting point for adapting a
  systematic mining process.
- **Excursion relation to EoI:** This mission is an Excursion of
  `M-expressions-of-interest.md`. HEAD §6 makes the unresolved point
  explicit: missions seem to be, imply, or metabolize EoIs, but the
  relation is not yet clearly articulated. That clarity-gap is in scope
  as a mission-coherence question, though not necessarily for full
  closure in IDENTIFY.
- **Mission as shareable practice:** HEAD §5 anchors the outward-facing
  importance. If mission practice can be made explicit, it becomes
  something Joe could explain, publish, and potentially distribute
  beyond the futon stack.

## 2. MAP

### Survey premise

The mission corpus should be accessed first through existing
aggregation surfaces rather than through raw filesystem discovery
alone. This is both methodological and substantive. Methodologically,
it lets the mining start from surfaces where the stack has already made
missions legible to itself. Substantively, those surfaces may already
contain partial theories of missionhood, cross-mission linkage,
portfolio shape, or mission-as-domain formalization that the
mission-coherence library should absorb rather than rediscover.

Accordingly, the corpus for MAP is not defined in the first instance
as "all `M-*.md` files on disk." It is defined as the mission material
that is already *made accessible* through existing aggregation
surfaces. Raw directory discovery remains useful, but only as a later
reconciliation pass: it can reveal omissions, blind spots, or
surface-specific bias, but it should not be the first act of corpus
construction.

For this mission, the initial aggregation surfaces are:

- `futon3c/holes/missions/M-war-machine.md`
- `futon3/holes/missions/M-live-geometric-stack.md`
  (Joe's "M-stack-geometry" reference)
- `futon3c/holes/missions/M-mission-peripheral.md`

These are not assumed to be exhaustive. They are the first surveying
surfaces through which the corpus should become reachable.

### Access rule for the mission corpus

For this mission's MAP phase, "Mission Corpus" means:

1. The set of missions and mission-adjacent coordination surfaces that
   are reachable through named aggregation surfaces already operating
   in the stack.
2. The union of those reachable sets, with provenance preserved per
   surface rather than collapsed immediately into one flat list.
3. Only after that, a reconciliation layer from raw repository scans
   to identify what existing aggregation surfaces fail to expose.

This matters because the mining target is not only the missions
themselves, but also the *forms of legibility* under which the stack
already gathers missions together.

### Aggregation surfaces — initial reading

#### 1. War Machine

`M-war-machine.md` is an aggregation surface in the strategic /
integrative sense. It is not just one mission among others; it is a
surface where multiple stack-level concerns are synthesized and where
cross-mission relations become part of a larger operational body. MAP
should treat it as a place where the *function* of missions in the
stack's strategic metabolism may already be partially articulated.

#### 2. Live Geometric Stack / "stack geometry"

`M-live-geometric-stack.md` is an aggregation surface in the substrate /
cross-codebase sense. It already carries dense cross-references across
missions and repos, including reflections on `M-war-machine`. MAP
should treat it as evidence for how missions appear when the stack is
viewed as a connected geometric object rather than as isolated work
items.

#### 3. Mission Peripheral

`M-mission-peripheral.md` is an aggregation surface in the formal /
machinic sense. It explicitly treats missions as a domain with a cycle
machine, tool gates, evidence emissions, and mission-state snapshots.
MAP should treat it as prior art for the claim that missions are not
mere markdown notes but structured operating objects.

### Immediate inference from Q1

The three named aggregation surfaces do not aggregate the corpus in the
same way:

- **War Machine** aggregates missions as strategic signals inside a
  larger body concerned with synthesis, triage, and continuous
  assessment.
- **Live Geometric Stack** aggregates missions as linked structural
  objects inside a cross-codebase substrate.
- **Mission Peripheral** aggregates missions as a domain to be executed,
  gated, and evidenced by a cycle machine.

That difference is already a mining clue. Candidate
mission-coherence patterns are likely to live not only *within*
missions, but also in the way missions become legible under these
different aggregation regimes.

It also sharpens the corpus question. If the mission corpus is meant to
be accessible from existing aggregation surfaces like these, then MAP
should test two things separately:

- **Reachable corpus:** what a user of the existing stack can already
  get to through strategic, geometric, and machinic aggregation.
- **Residual corpus:** what still exists as mission material on disk
  but is *not* yet gathered by those surfaces.

The residual is not noise. It is evidence about the current limits of
mission legibility inside the stack.

### Q5 — operational first mining pass

The practical question for this session is not whether the aggregation
surfaces perfectly agree. It is whether mission-pattern mining is
operationally possible on a corpus that is small enough to handle and
large enough to yield recurring structure.

The method note at `~/Downloads/1996461.1996523.pdf` suggests an
important adaptation rule: the mining unit should be *issues* extracted
from source material, not documents counted one by one. In the paper's
setting, a data pool of fewer than roughly 150 issues is too small. For
mission mining, this suggests a feasible first pass if a modest corpus of
rich mission documents can yield an issue-pool of that order.

#### Proposed first-pass corpus

Use a corpus of approximately 8-12 rich mission documents, selected
through the named aggregation surfaces and biased toward missions that
already contain substantial self-description, relation-structure, and
claims about how the work proceeds.

Initial candidate slice:

- `M-war-machine.md`
- `M-live-geometric-stack.md`
- `M-mission-peripheral.md`
- `M-self-representing-stack.md`
- `M-three-column-stack.md`
- `M-pattern-application-diagnostic.md`
- `M-stack-inhabitation.md`
- `M-portfolio-inference.md`
- `M-a-sorry-enterprise.md`

This is large enough to span multiple mission styles and stack roles,
but still small enough for a manual or semi-manual first mining pass.

#### Mining operations for the first pass

For each mission in the first-pass corpus:

1. Extract mission issues in the broad sense used by the method note:
   recurring problems, decisions, solutions, operating constraints,
   evidence disciplines, and self-descriptions of what the mission is
   doing.
2. Normalize those issues into a worksheet so that near-duplicates can
   be grouped across missions.
3. Compute recurrence by *mission-level occurrence*, not paragraph
   frequency, so one verbose mission does not dominate the counts.
4. Mark whether each recurring issue is supported by:
   explicit self-description, repeated structure, contrast case,
   failure mode, or follow-on dependency.
5. Promote only those candidate patterns that recur across multiple
   missions and carry enough evidence to say more than "this sounded
   good once."

#### Feasibility criterion for MAP

The first mining pass is feasible if:

- the 8-12 document slice can be extracted into a shared issue-pool
  without explosive manual overhead, and
- that pool yields a non-trivial set of recurring candidates rather than
  a grab-bag of one-off observations.

If that criterion fails, the mission should stop and say so explicitly:
either the corpus slice is too small, the extraction operation is too
vague, or the missions are not yet written in a form that supports
pattern mining.

#### Pilot result: 3-mission feasibility pass (2026-05-11)

A pilot pass has now been run against the three aggregation surfaces
named above:

- `M-war-machine.md`
- `M-live-geometric-stack.md`
- `M-mission-peripheral.md`

Artifacts:

- `futon3/library/mission-coherence/PILOT-WORKSHEET-2026-05-11.md`
- `futon3/library/mission-coherence/PILOT-SUMMARY-2026-05-11.md`

The pilot result is positive. The extraction operation produced a
coherent issue-pool with recurring cross-mission clusters, which is
enough to say that mission-pattern mining is operationally viable.
However, the pilot is still only a feasibility pass. It validates the
worksheet shape and the recurrence test; it does not yet supply enough
material for the final pattern library on its own.

Operational consequence: proceed to a wider first pass using the same
worksheet shape, preferably at the upper end of the planned corpus
(roughly 10-12 rich missions) unless the extraction density is
substantially increased.
